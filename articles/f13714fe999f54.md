# はじめに

Unityプロジェクトに広告SDKやユーザーデータ分析系SDKを導入すると、**Androidビルドがフリーズする／解決に難航するGradleエラーが出る**といったトラブルに遭遇することがあります。

今回は、**Unity 6 + Facebook SDK導入後にAndroidビルドが進まなくなった問題**について、実際に辿った調査の流れを時系列順に整理しつつ、原因と対処をまとめます。

同様のエラーに悩まされている方にとって「何を疑うべきか」「どこを見ると突破口になるか」の参考になれば幸いです。

## 環境

- OS: macOS Sequoia 15.6.1
- Unity: 6.0 (6000.0.62f1)
- Facebook SDK for Unity: 18.0.0
- Android Build Support / OpenJDK: Unity Hub経由でインストール済み

https://github.com/facebook/facebook-sdk-for-unity/releases/tag/sdk-version-18.0.0

# 発生した問題

Unity 6のプロジェクトにFacebook SDKを導入後、Android向けにビルドを実行すると

- `Building scene 0...` から先に進まない
- Unity Editorがフリーズしたような状態になる
- 明確なビルドエラーダイアログが出ない

という状況に陥りました。

ログを確認すると、以下のようなエラーが出力されていました。

```shell
Gradle failed to fetch dependencies.

stderr:
The operation couldn't be completed. Unable to locate a Java Runtime.
Please visit http://www.java.com for information on installing Java.

exit code: 1
```


## 調査①：本当にJava Runtimeが存在しないのか？

ログだけを見ると「Java Runtimeが見つからない」ように見えますが、Unity Hub経由でAndroid Build Supportをインストールしている場合、Unityは **同梱のOpenJDK（Unity 6ではJava 17）** を使用します。


![](https://storage.googleapis.com/zenn-user-upload/331d4e293575-20251126.png)

そのため

- Java自体が未インストール
- JAVA_HOME が未設定

といった単純な問題ではなさそうだと判断しました。

この時点で

> 表面的には Java Runtime エラーに見えるが、  
> 実際には **Gradle側の起動に失敗しているだけでは？**

という疑いを持つようになりました。

## 調査②：初動の切り分け

次に、一般的な問題の切り分けを行いました。

- Unity Editorの再起動
- Libraryフォルダ削除
- Clean Build
- Facebook SDKの初期化コード（`FB.Init()` など）の一時削除

しかし、状況は変わりません。

続いて、External Dependency Manager（EDM4U）による依存解決を疑い、

- `Assets > External Dependency Manager > Android Resolver > Force Resolve`

を実行しましたが、こちらもResolver実行中に同様のエラーで停止しました。

### 補足

Settingsの`Enable Auto-Resolution`チェックを外すと、Resolveの自動実行は無効化できます。

![](https://storage.googleapis.com/zenn-user-upload/d14cd0ed1775-20251126.png)

ここで初めて、Java Runtime そのものではなく**Gradle/Groovy の初期化段階で失敗している**ことが明確になりました。


## ログから分かったこと

実際のログを整理すると、以下の点が読み取れます。

- エラーは Unity本体のAndroidビルドではなく`Temp/PlayServicesResolverGradle/gradlew` による**依存関係ダウンロード処理** で発生している
- `org.codehaus.groovy.vmplugin.v7.Java7` の初期化に失敗している
- `PlayServicesResolver`（= EDM4U）が内部で使用している Gradle/Groovy が起動段階で落ちている

この時点で

> Javaが存在しない  
> → ❌  
>
> Java実行環境と、EDM4Uが内部で使用している  
> Gradle / Groovy の互換性問題  
> → ⭕️  

という切り分けができました。

# 原因の分析

ここで、本事象に関わる開発環境のバージョン情報を整理します。

- Unity 6 に同梱されている Java 実行環境  
  - **OpenJDK 17**
- Facebook SDK for Unity  に同梱されていた External Dependency Manager  
  - **EDM4U v1.2.166**
- 上記 EDM4U が内部で使用している Gradle  
  - **Gradle 5.1.1**
- 本記事執筆時点（2025-11）での EDM4U 最新版  
  - **EDM4U v1.2.186**（2025年5月リリース／その後 `v1.2.187` が2026年1月にリリース）

## 詳細確認

さらに詳細を見ていきます。

### Unity 6は **Java 17** を前提としている

2022.3 LTSでは `Java 11` 前提でしたが、Unity 6より `Java 17` サポートとなりました。

|Unity version|JDK version|
|---|---|
|6000.0+|17 (OpenJDK version 17)|
|2022.3 LTS|11 (OpenJDK version 11)|
|2021.3 LTS|11 (OpenJDK version 11)|

https://docs.unity3d.com/6000.2/Documentation/Manual/android-supported-dependency-versions.html

### Facebook SDK同梱のEDM4Uが古い

`Facebook SDK 18.0.0`には、2021年6月リリースの `v1.2.166` がバンドルされています。

https://github.com/facebook/facebook-sdk-for-unity/releases/tag/sdk-version-18.0.0

https://github.com/googlesamples/unity-jar-resolver/releases/tag/v1.2.166

### EDM4Uが利用するGradle 5.1.1がJava 17環境で初期化に失敗する

EDM4U`v1.2.166`で使用されるGradleバージョンは、2019年1月リリースの`Gradle 5.1.1` です。

https://gradle.org/releases/#5.1.1

Gradle公式の互換性表によると、Gradle自体をJava 17上で実行できるのは **Gradle 7.3以降** とされています。Gradle 5.1.1はそもそもJava 17を想定していない世代のため、Unity 6が使用するJava 17環境では、Gradle/Groovy の初期化段階で `org.codehaus.groovy.vmplugin.v7.Java7` の `NoClassDefFoundError` が発生します。

https://docs.gradle.org/current/userguide/compatibility.html

注意したい点として、**`v1.2.186`（2025年5月リリース）や `v1.2.187`（2026年1月リリース）でも、Gradle Wrapperが参照しているGradleは依然として `5.1.1` のまま** です。つまりこれは「`v1.2.166` が古いから Gradle が古い」というよりも、**EDM4U全般が抱える Gradle/Java 互換性の問題** として捉えるのが妥当です。

この構成を整理すると

- Java: 17（比較的新しい）
- Gradle / Groovy: 5.1.1 世代（かなり古い）

という **世代差の大きい組み合わせ** によって、EDM4Uの依存解決処理が成立しなくなっていました。


要するに、今回の問題の本質は

**「Javaが見つからない」ことではなく、EDM4U / PlayServicesResolverGradle が実行するGradle 5.1.1がUnity 6のJava 17環境で初期化に失敗していたこと、加えてFacebook SDK同梱の古いEDM4Uや複数バージョンのEDM4U混在が、この問題を顕在化させる要因となっていた**

点にあります。


# 対応策

今回実施した対応は以下です。

- Facebook SDKに含まれる不要なExamplesを削除
- SDKに同梱されていたEDM4U / PlayServicesResolverを削除
- プロジェクト側でEDM4Uを最新版（`v1.2.186`）に揃える
- `Force Resolve` を再実行

この対応により、EDM4Uの重複や古いresolverの混在が解消され、Androidビルドはフリーズせず依存関係の解決が正常に通るようになりました。

https://github.com/googlesamples/unity-jar-resolver/releases/tag/v1.2.186

:::message
前述のとおり、`v1.2.186` / `v1.2.187` でも Gradle Wrapper 自体は `5.1.1` を参照しているため、「EDM4U の更新だけで Gradle 5.1.1 と Java 17 の互換性問題が解消する」とは限りません。

今回の改善要因は、SDK同梱版との混在解消、不要な依存解決対象の削除（Examples 削除）、`Force Resolve` による `Assets/Plugins/Android` 側の再生成など、**複数の要因が複合的に効いた**可能性が高いです。
:::

Facebook SDKに限らず、広告SDKやユーザーデータ分析系SDKの多くは、それぞれが独自にEDM4Uをバンドルしています。

その結果

- プロジェクト内に複数バージョンのEDM4Uが存在する
- どのResolverが実行されるか不定になる
- 古いGradle / Groovy が意図せず使われる

といった状況が発生し、今回のようなエラーにつながりやすくなります。

## 複数SDK導入時の注意点

特に重要だったのは

**「SDK任せにせず、EDM4Uをプロジェクト側で明示的に管理する」**

という点でした。

広告SDKやユーザーデータ分析系SDKを複数同時に導入する場合、  それぞれにバンドルされたEDM4Uがプロジェクト内に複数存在し、依存関係解決の不安定さを招くことがあります。

EDM4Uは設計上、**1プロジェクトにつき1つだけ存在する前提**で動作します。

そのため実務上は

- 各SDKに同梱されたEDM4Uは使用せず
- プロジェクト内に最新版のEDM4Uを1つだけ配置し
- それを全SDK共通で参照する

という運用が、事実上のベストプラクティスとして採用されているようです。

```shell:ex.)
Assets/
├─ ThirdParty/
│  ├─ ExternalDependencyManager/   ← 最新版EDM4U
│  ├─ FacebookSDK/
│  ├─ Firebase/
│  ├─ AdMob/
│  └─ [OtherSDK]/
```

# まとめ

今回の問題は「Facebook SDKの不具合」というよりも

**複数SDK × 古いEDM4U × 新しいUnity（Java）環境**

という組み合わせによって顕在化したものでした。

特に、最初に表示された「Java Runtimeが見つからない」というエラーはミスリーディングで、**調査を進める中でログの変化を追うことが重要**だと、あらためて感じました。

具体的な原因の特定と問題解決にだいぶ苦戦したので、同じようなところで詰まっている方の助けになれば幸いです。
