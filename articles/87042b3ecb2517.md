# LT1: LLMは障害対応のお友達（[@sashimimochi343](https://x.com/sashimimochi343)）

@[speakerdeck](b7483787f0f74788a0ca19220ecbbb67)

- 深夜のアラート対応など、エンジニアの負担となるシステム障害時の **「検索疲れ」** を LLM の力で解消する取り組みを紹介
- AI に「エラーが発生した」と雑に質問するだけで、過去の類似障害や対処法を自動で調べ、ログやナレッジベースを横断して回答してくれる仕組みを構築
- 仕組みの中核には、LLM と外部ツールを標準的な方法で接続する **[MCP（Model Context Protocol）](https://modelcontextprotocol.io/)** サーバーを採用
- MCP の導入により、AI が自ら検索クエリを考えて情報収集できるようになり、人手で複数のツールを行き来する手間を削減
- 結果として、情報が分散しがちな緊急時の対応が手軽になり、**新人でもオンコール対応に参加しやすくなるような仕組み** が実現できる、という展望が語られた

# LT2: AIファーストな開発プロセスへ：AI駆動開発の核心を解説する（[@eltociear](https://x.com/eltociear)）

@[speakerdeck](ce4aaf4883f34235a845ed8ff65a66d5)

- 生成 AI を前提とした新たな開発スタイル **「AI 駆動開発」** の原理や実践方法、今後の展望を解説
- AI の強みは、要件から構造を理解し、全体的な整合性を保ちながらコードやドキュメントを自動生成・編集できる点にあり、人間は **問題設定や最終判断といった上流工程に集中** する役割分担が重要
- アーキテクチャ面では、IDE 常駐 AI と MCP の組み合わせにより、AI がローカルファイルや DB に直接アクセスし、開発プロセス全体に組み込まれる構造が紹介された
- 実践方法として、AWS が提唱する **[AI-DLC（AI-Driven Development Life Cycle）](https://aws.amazon.com/blogs/devops/ai-driven-development-life-cycle/)** を取り上げ、開発スピードや品質が飛躍的に向上することが示された
- 将来的には、少人数で大規模なプロダクトを開発できる **「1人ユニコーン」** の時代が来る、という大胆な予測で締めくくられた

# LT3: Androidアプリの中断はすげー難しい（[@fkm](https://x.com/fkm)）

- Android アプリがバックグラウンドに回った際の **「中断と再開」** に関する処理の難しさを、実演を交えて解説
- AI に書かせた一見正しく動くコードでも、Android の [開発者オプション](https://developer.android.com/studio/debug/dev-options) で **「アクティビティを保持しない」** をオンにしてテストすると、アプリ再開時にカウントなどの変数がリセットされてしまう問題を実演
- これを防ぐためには [`onSaveInstanceState`](https://developer.android.com/topic/libraries/architecture/saving-states) などを用いた **状態の保存・復元処理** が必要になる
- さらに **「バックグラウンドプロセスを使用しない」** 設定をオンにすると、プロセス自体が停止し、シングルトンで保持していたログイン状態のような変数まで初期化されてしまう
- 対策として、メモリ上に状態を置くのではなく [SharedPreferences](https://developer.android.com/training/data-storage/shared-preferences) や [DataStore](https://developer.android.com/topic/libraries/architecture/datastore) などのストレージに保存するよう設計を見直す必要がある
- [Android のアクティビティ・ライフサイクル](https://developer.android.com/guide/components/activities/activity-lifecycle) 特有の難しさと、開発者オプションを活用した **適切なテストの重要性** が語られた
