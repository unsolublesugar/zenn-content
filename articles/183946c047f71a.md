# LT1: Android-DevPortal作ってみたよ（[@fkm](https://x.com/fkm)）

https://docs.google.com/presentation/d/1wzjtnxN9q59GYb0AxpmizGJV3xQ_GSmTDF-Gb87XLbU/edit?usp=sharing

- Androidアプリ開発におけるデバッグ画面は便利な反面、実装の手間・ビルド時間への影響・本番への混入リスク（誤出荷）が課題
- 解決策として「**アプリ本体と同じプロセスで動きつつ、別のアプリアイコンとして起動するデバッグ用Activity**」を実装
- `AndroidManifest.xml` の `<intent-filter>` に [`action.MAIN`](https://developer.android.com/guide/topics/manifest/action-element) と [`category.LAUNCHER`](https://developer.android.com/guide/topics/manifest/category-element) 、さらに [`taskAffinity`](https://developer.android.com/guide/components/activities/tasks-and-back-stack) を設定し、別ランチャーアイコンから起動可能に
- ルーティングは Google 公式の [Navigation 3](https://developer.android.com/guide/navigation/navigation-3) のレシピと、DI コンテナ [Hilt](https://developer.android.com/training/dependency-injection/hilt-android) の [`@IntoSet`](https://dagger.dev/dev-guide/multibindings.html)（特定の型のインスタンスを1つの Set に集約する機能）で完全モジュール化
- `dependencies` に `debugImplementation` として追記するだけで、In-app Billing の確認や UI 確認といったデバッグ機能を着脱可能。不要になれば記述を消すだけで完全除去できる
- 開発を楽にするライブラリ「**[Android-DevPortal](https://github.com/mokelab/android-devportal)**」として OSS 公開

# LT2: ライブ配信技術のこれまでとこれから HLS/DASH, WebRTC, Media over QUIC（[@mti_daiki](https://x.com/mti_daiki)）

@[speakerdeck](225e4cea98f940a49269985c1a7f4829)

- 既存の映像配信技術が抱える「**低遅延**」と「**大規模配信（スケーラビリティ）**」の両立というジレンマを整理
- 既存技術の限界:
    - **[HLS](https://developer.apple.com/streaming/) / [DASH](https://dashif.org/)**: セグメント化と HTTP GET で CDN に乗ってスケールするが、構造的に 10〜20 秒の遅延が避けられない
    - **[LL-HLS](https://developer.apple.com/documentation/http-live-streaming/enabling-low-latency-http-live-streaming-hls)**: 部分セグメント等で 2〜5 秒まで遅延を改善したが、依然として HTTP のリクエスト/レスポンスの枠内に留まる
    - **[WebRTC](https://webrtc.org/)**: 極めて低遅延だが、既存の CDN インフラに乗せられず大規模スケールが難しい
- 次世代プロトコル **[Media over QUIC (MoQ)](https://datatracker.ietf.org/wg/moq/about/)**: 既存技術のいいとこ取りを目指し、現在 IETF で策定中（ドラフト段階）
- MoQ の仕組み: [QUIC](https://datatracker.ietf.org/doc/html/rfc9000) の上に Pub/Sub モデルを構築。視聴者が `SUBSCRIBE` するとフレーム単位で順次データが流れてくるため低遅延を実現しつつ、中継サーバー（Relay）を [Cloudflare](https://www.cloudflare.com/) などの既存 CDN インフラに乗せることで大規模ファンアウトが可能

# LT3: ますます助かる障害対応自動化（[@sashimimochi343](https://x.com/sashimimochi343)）

- AI エージェント（[MCP](https://modelcontextprotocol.io/) サーバーなど）を活用し、深夜のシステム障害対応を自動化・効率化する取り組みの続編。**技術書典の新刊内容として執筆中**
- ログ分析だけでなく **メトリクス（CPU 使用率・メモリ・ネットワーク帯域）** も AI が自律的に調査できるようシステムを拡張。AI が自らサーバーにアクセスしてクエリを書き、ダッシュボードを生成して人間に必要な情報を届けてくれる環境が実現しつつある
- 技術的には障害復旧の **完全自動化（レベル4）**（[自動運転の自律性レベル](https://www.mlit.go.jp/jidosha/anzen/01asv/resourse/data/levels.pdf)になぞらえた区分で、人間の介入を要さない最上位段階）が視野に入ってきたが、「システムがやらかした際の責任を誰が取るのか」という観点から本番環境への無人適用にはまだ懸念が残る
- 結論: ログ・構成図・対応履歴を AI がアクセスしやすいよう整備することは、**「新しく入った人間のオンボーディングに必要な環境（やらかしても大丈夫な仕組み）」を整えることと同義**
