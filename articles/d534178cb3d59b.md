# LT1: リファクタは始まる前に終わっている そして、AI時代でも変わらない理由（[@paseri_kurosawa](https://x.com/paseri_kurosawa)）

@[docswell](https://www.docswell.com/s/3822049/ZVJ47V-2026-05-19-233358)

- リファクタリングを単なる「コードをきれいにする作業」ではなく、明確な目的を持った **「構造改善活動」** と再定義し、事前の計画や設計こそが成否を分けると説いた発表
- リファクタリングは **「調査・整地・剪定・再編・検証・収束」の6フェーズ** で成り立っており、それぞれの工程で達成すべき目的や視点が異なる、というフレームワークが提示された
- 中でも実装前の前工程である **「調査・整地・剪定」の段階で成否がほぼ決定する** と主張し、いきなりコードに手を入れるのではなく、構造の理解と方針整理に最も時間を割くべきと強調。また、変更単位を小さくし、テストやリリースを細かく分ける **「細分化デプロイ（リスクの分割払い）」** の重要性も語られた（※[Zenn記事](https://zenn.dev/paseri_kurosawa/articles/80c6107b443e1d)に記載）
- AI時代においても、**「コンテキストウィンドウを超えると全体像が維持できない」「『触ってはいけないコード』などの暗黙知や文化的文脈を観測できない」** といった壁が存在するため、前工程における「全体像の維持」や「境界線の判断」はAIには難しく、依然として人間が責任を持って担う必要があると指摘
- 一方でAIは、人間が設計した計画に沿って「再編」フェーズ（命名の改善やモジュール分割など）を担う **強力なアシスタント** として機能するため、「前工程は人間が・実行はAIが」という役割分担の本質が解説された発表だった

https://zenn.dev/paseri_kurosawa/articles/80c6107b443e1d

# LT2: SaaSの権限管理設計を完全に理解した（[@rsugi8](https://x.com/rsugi8)）

@[speakerdeck](3f8874235ad748d7af0df119d74643c5)

- マルチテナント SaaS において複雑化しやすい「権限管理」の課題と、その解決策となるアーキテクチャについて解説した発表
- ロールベースの条件分岐（例：`if user.role === "admin"`）をコードへ直書きしたり、権限用の謎テーブルを増殖させてしまうといった **「やらかしパターン」の限界** を、実例を交えて指摘
- これを解決するための設計として、**「何ができるか」をPBAC（ポリシーベースアクセスコントロール）としてポリシークラスに集約** し、ビジネスルールをコードの一箇所に閉じ込めるアプローチを提案。この仕組みにより、「ロール」だけでなく **「加入プラン（pro、basicなど）」の違いによる機能制御も組み合わせて評価できる** ため、`shop_owner_pro` のような謎ロールが増殖する問題も防ぐことができる
- さらに、テナント間の横断アクセスなどを防ぐために **「誰のリソースか」をReBAC（リレーションベースアクセスコントロール）で管理・検証する** という、2層に分けた設計が紹介された
- ReBAC では、権限がないリソースへのアクセスに対して403（権限エラー）ではなく **404（Not Found）を返すことでリソースの存在自体を隠蔽し、セキュリティリスクを下げる** というベストプラクティスも共有（※[Zenn記事](https://zenn.dev/rsugi/articles/19e0014a7040b3)に記載）
- 「何ができるか」と「誰のリソースか」の関心を分離することで、機能追加や仕様変更にも壊れにくい権限管理の仕組みを実現できる。サンプルプロジェクトのデモも交え、わかりやすく整理してくれた発表だった

https://zenn.dev/rsugi/articles/19e0014a7040b3
