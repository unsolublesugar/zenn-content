## はじめに

国内株・米国株・投資信託・金（ゴールド）と、資産が複数の口座に散らばっていると、「全体で今いくらで、どんな構成で、どれだけ増えているのか」を一目で把握するのが意外と難しいです。証券会社のアプリは口座ごとに分断されているので、資産全体をひとつの構成として俯瞰する視点が持ちにくい。

残高を確認するたびにアプリを開いてログインし直すのも地味に面倒だし、開けば開いたで、自分が興味のないキャンペーンやおすすめ銘柄といったノイズが目に入る。そういう余計なものを取り払って、自分の資産だけを静かに眺められる場所がほしい、というのが出発点でした。

もうひとつ、公開してみたいという気持ちもありました。新NISAをきっかけに投資に興味を持つ人が増えて、成功者のポートフォリオを紹介する動画なんかもよく見かけます。ただ中身は「国内株が何割、米国株が何割、債券や金も少々」くらいのざっくりした話か、億り人の参考にしづらい持論が多い印象でした。

具体的な保有数量や、割合の変遷、投資額の推移まで、シンプルかつオープンにする人がもっといてもいいのでは、と前から思っていたので、まずは自分が先陣を切ってみることにしました。

昔から投資系ブログでこうした情報を公開している人はいましたが、どうにも無骨で、パッと見では参考にしづらい。パッと見られて、画面を触りながら「お、この銘柄をこれだけ持ってる理由は何だろう？」と、見る人の興味をちょっと刺激するような形にできないか。それに適したかたちを模索した結果が、このプロダクトです。

もちろん家計簿アプリや証券アプリでも、複数のアセットを総合的に見ること自体はできます。ただ、API連携やアカウントログインの煩わしさがあったり、前述のノイズが多かったり、自分好みにカスタマイズしづらかったりする。そのあたりの煩雑さを排して、運用コストも最小限に抑えたデータの取得・更新・公開のやり方を、けっこう真面目に考えました。

元手は30万円ほどから始めて、気づけば5年くらいの運用実績が溜まっていました。ちょうど証券口座に置いている資産が1,000万円を超え、評価損益も+300万円前後に達したところで、これくらいの期間と規模になれば、それなりに参考になるものもあるかもと思い、開発と公開を進めることにしました。

そこで、自分の保有資産を一枚で見渡せるダッシュボードを作りました。

この記事で書きたいのは、作り方の段取りの話です。今回は次の二段構えで進めました。

- **Claude Design** … ダッシュボードの大枠のデザインと、動くプロトタイプを作る。配色・レイアウト・コンポーネントの方向性をここで固める。
- **Claude Code** … プロトタイプを実データで動く本番へ。内部実装のリファクタ、データ設計、CI、そして実機で見て気になった所の細かな挙動調整をやる。

デザインの大枠は Claude Design でさくっと、細部の作り込みは Claude Code で、という分担が思いのほか気持ちよくハマったので、その利用事例として、技術選定の理由・データ設計の考え方・何を足して何をやめたかの試行錯誤を残しておきます。

:::message
**Claude Design とは**：Claude 上で、自然言語の指示からデザインのプロトタイプを生成・編集できる機能です。配色・レイアウト・コンポーネントを含む「動くUI」を複数案で出し、ブラウザ上で見比べながら方向性を固められます。

- [Claude Designでデザインシステムをセットアップする](https://support.claude.com/ja/articles/14604397-claude-design%E3%81%A7%E3%83%87%E3%82%B6%E3%82%A4%E3%83%B3%E3%82%B7%E3%82%B9%E3%83%86%E3%83%A0%E3%82%92%E3%82%BB%E3%83%83%E3%83%88%E3%82%A2%E3%83%83%E3%83%97%E3%81%99%E3%82%8B)
:::

🔗 **<https://unsolublesugar.github.io/investment-portfolio/>**

完成したものの主な機能はこんな感じです。

- **資産構成ドーナツ（円グラフ）＋凡例** — 区分（国内株・米国株・投資信託・金）別の評価額・構成比・評価損益。カテゴリを選ぶと銘柄内訳へドリルダウン。中央が空いた円グラフを本記事では「ドーナツ」と呼びます。
- **資産推移グラフ** — 評価額の推移（1年/3年/5年/全期間）と、資産構成の積み上げ棒（月次/年次）。日次スナップショット履歴から描画。
- **銘柄テーブル** — 銘柄ごとの現在値・前日比・前月比・評価額・評価損益。PCはヘッダクリック、モバイルはチップで並び替え。
- **損益率ランキング** — 評価損益率の上位3／下位3銘柄。
- **自動更新** — 国内株・米国株の時価は GitHub Actions が毎日 Yahoo Finance から取得。為替（USD/JPY）と金スポットはブラウザからライブ取得（いずれもキー不要API）。
- **OGP画像の自動生成** — SNSシェア時に、その日の資産構成ドーナツと区分別の割合・損益が入ったカード画像が出る。

## Claude Design で動くプロトタイプを作る

最初の一歩は Claude Design でした。やりたいことの大枠──「複数口座に散った資産を一枚で見たい」「証券アプリっぽい落ち着いたトーンで」「ドーナツで構成比、下に明細」くらいの解像度──を言葉で伝えると、配色からレイアウト、コンポーネントの粒度まで含めた動くプロトタイプが返ってきます。しかも嬉しかったのが、方向性を1案に絞らず複数出してくれるところでした。

このときは次の3案が上がってきました。

- **案A クラシック** … カード＋極薄影で情報を区切る、証券ダッシュボード然とした正統派
- **案B カード** … 区分ごとに大きめのカードを並べる構成
- **案C 一覧** … 表ベースで密度を上げた構成

実物を並べて見比べられるので、「これは自分の用途だと密度が高すぎる」「これは余白が気持ちいいけどスクロールが長い」といった判断が、文字の仕様書を読むよりずっと速いです。最終的に案A クラシックを採用しました（案B・案C の撤去は、後で Claude Code に頼んでいます）。

### デザインの方針：色は意味にだけ使う

Claude Design とのやりとりで固まったトンマナ（トーン＆マナー。配色や装飾の一貫した方針のこと）が、楽天証券 iGrow 寄りの「明るい白基調・低彩度・情報密度高め」です。装飾は最小限で、色は意味づけ（カテゴリ識別・騰落）にだけ使う、というルールをここで言葉にできたのが大きかった。

- **騰落色は日本式** … プラス＝赤、マイナス＝青、変化なし＝グレー。証券アプリに慣れた目に素直です。ドーナツ中央・サマリー・テーブル・ランキングまで全画面で一貫させます。
- **カテゴリ色の一貫運用** … 国内株・米国株・投信・金の4色を、ドーナツ・凡例・積み上げ棒・明細のアクセントで同一色に結び、視線移動のコストを下げます。
- **飾りの色は足さない** … グラデーションを使うのはロゴだけ。きれいに飾るより、意味のない色・数字・装飾を足さない方向で密度を上げる、という考え方です。

この方針がプロトタイプの段階で乗っていたおかげで、後の Claude Code フェーズで何度も細部をいじっても、判断の軸がぶれませんでした。新しい要素を足すたびに「これは意味のある色か？」と問い直せます。デザインの羅針盤を先に作るという意味で、Claude Design はこの手のプロジェクトと相性が良かったです。

## Claude Code に持ち込んで作り込む

プロトタイプはあくまで第一稿です。実際の保有データを流し、CIで毎日更新し、スマホでも崩れないようにして、ようやく毎日見る道具になります。ここからは Claude Code に持ち込みました。

最初の依頼はこの2つでした。

1. 「案A クラシックを採用するので、案B・案C 用の実装を削除して」── プロトタイプの分岐を畳んで本筋に絞る。
2. 「PC表示とスマホ表示の中間くらいの横幅で、ヒーロー（画面最上部の主役表示領域）の評価損益テキストがはみ出ることがあるので直して」── 新機能ではなく、作りかけの取りこぼしから潰す。

派手な新機能より、まず今あるものを正しく動かす。この地味な2手から本番化が始まりました。以降はほぼすべて Claude Code フェーズの記録です。

## 技術スタックの選定

本番化にあたって最初に決めたのが土台です。このダッシュボードは保有資産を閲覧するだけで、ログインも書き込みもいりません。データは自分で管理します。そこで、フレームワークを使わずビルド工程のない静的サイトにしました。

- **ランタイム**: dc-runtime（`support.js`）の `<x-dc>` テンプレート＋薄い Component
- **言語**: Vanilla JS（依存パッケージなし）
- **スタイル**: 素の CSS（`pf-` プレフィックスのクラス）
- **ホスティング**: GitHub Pages
- **CI/CD**: GitHub Actions（株価取得・スナップショット・OGP生成・デプロイ）
- **外部API**: 為替・金・株価いずれもキー不要のものを採用

:::message
**dc-runtime** は、`<x-dc>` というカスタム要素でテンプレートと `Component` クラスを結びつけ、ビルド工程なしで画面を組み立てる軽量ランタイムです。今回は Claude Design で作ったプロトタイプに、その生成物である `support.js` が同梱されていたので、それをそのまま土台として使っています。`support.js` 自体はビルド生成物で、ファイル冒頭にも `// GENERATED from dc-runtime/src/*.ts — do not edit.` と明記されています。そもそも「ランタイム」とは何かについては [JavaScriptのランタイムについて知らなさ過ぎたので調べた](https://zenn.dev/tenchijin/articles/20240618_runtime) が分かりやすいです。
:::

:::message
**Vanilla JS** は、ライブラリやフレームワークを使わない「素の JavaScript」を指す呼び方です。なぜそう呼ぶのかは [なぜ素の JS のことを Vanilla と呼ぶのか調べてみた](https://zenn.dev/seya/articles/66809b3d59151c) が詳しいです。
:::

一番大事にしたのは、メンテナンスコストを極力抑えることです。「数年後の自分がメンテできる」というより、そもそもメンテにかかる手間を最初から小さくしておきたい、という気持ちのほうが強い。個人で長く使う道具では、ここが効いてきます。フレームワークのメジャーバージョン更新、バンドラの設定、依存ツリーの腐敗。こうしたものは時間が経つほど、動かすまでの手間を重くします。

ビルドレスにしておけば、`python3 -m http.server` で開くだけ、`file://` で直接開いても動きます。CI が複雑な依存を抱える必要もありません。

```bash
python3 -m http.server 8000   # → http://localhost:8000/dashboard.dc.html
```

`dashboard.dc.html` が `./styles.css` と `./js/*.js` を相対パスで読み込むだけの構成なので、ローカルでも GitHub Pages でもそのまま動作します。

### ロジックを mixin で分離する

プロトタイプは1枚のHTMLに全部入りでした。これだと変更箇所の区分がしづらく、メンテしにくい。そこで Claude Code に「関係する部分をカプセル化して、追加・変更を把握しやすい形にリファクタして」と頼みました。

ランタイムの制約上、Component 本体はインラインの `<script>` に置く必要があります（外部 `src` に切り出すと textContent が空になってしまう）。とはいえ全ロジックをHTMLに詰めると見通しが悪い。そこで実装の大半を `js/*.js` に出し、`window.PortfolioLogic` というメソッド集へ登録しておいて、Component の末尾で合成する形にしました。ここで言う mixin は、独立して定義したメソッド群を既存クラスに混ぜ込む手法のことです。

```js
// js/format.js などの末尾（冪等パターンで登録）
window.PortfolioLogic = Object.assign(window.PortfolioLogic || {}, {
  yen(n) { return '¥' + Math.round(n).toLocaleString('en-US'); },
  // ...
});

// dashboard.dc.html のインライン Component 末尾
Object.assign(Component.prototype, window.PortfolioLogic);
```

インラインに残すのは、ランタイムが名前で呼ぶ契約面だけです（`constructor`・ライフサイクル・`renderVals`）。これで見た目は `styles.css`、ロジックは `js/`、画面構造は `<x-dc>` テンプレート、という住み分けができました。この住み分けが、後半の「実機を見ては直す」往復を支えてくれます。

```text
dashboard.dc.html   アプリ本体（テンプレート＋薄いComponent）
styles.css          スタイル（pf- クラス）
js/
  format.js         整形（金額・%・色）
  data.js           取得・保存・既定保有データ
  model.js          集計（評価額・損益、ドーナツ/凡例、サマリー）
  chart.js          推移グラフのビューモデル
support.js          ランタイム（生成物・編集しない）
```

## CLAUDE.md とルールファイルでコンテキストを共有する

Claude Code との開発で効いたのが `CLAUDE.md` と `.claude/rules/` の整備です。プロジェクトの概要・コマンド・編集の住み分けを `CLAUDE.md` に書いておくと、毎回の会話で説明し直さなくても、前提を共有した状態で作業を始められます。

本プロジェクトの `CLAUDE.md` は、全体像の把握に必要な最小限だけを残し、詳細は `.claude/rules/` 配下のポインタで参照する構成にしています。

```markdown
# 資産ポートフォリオ ダッシュボード

必ず日本語で回答してください。

## Project
国内株・米国株・投資信託・金（ゴールド）の保有を可視化する公開ダッシュボード。
ビルド工程のない静的サイト（dc-runtime ベース）で、GitHub Pages にホスティング。

## 編集の目安
- 見た目 → styles.css（pf- クラス）
- ロジック → js/
- 画面構造 → dashboard.dc.html の <x-dc> テンプレート

## References
- アーキテクチャ: .claude/rules/architecture.md
- コーディング規約: .claude/rules/conventions.md
- データ保存方式: .claude/rules/data.md
- デザイン トンマナ: .claude/rules/design.md
```

ドメイン固有のルールは `.claude/rules/` に分離しています。たとえば「`dashboard.dc.html` に開発経緯やデータ保存方式を直接書かない」「`support.js` は生成物なので編集しない」「デザイン実値（`styles.css`／`js/format.js`）を変えたら `design.md` を同期する」といった規約です。Claude Design で固めたトンマナを `design.md` に落として、Claude Code がそれを正として参照する、という橋渡しにもなっています。

さらに、これらの規約はコミット前に自動検出するようにしています。pre-commit フック（`check-rules.mjs`）で、`support.js` の編集・HTMLコメントへの禁止語・絶対パス参照・デザイン実値変更時の `design.md` 未同期などを検査します。やってはいけないことを仕組みで止める、というつもりです。

```bash
git config core.hooksPath .githooks   # クローン後に一度だけ有効化
```

ルールを明文化して仕組みに組み込んでおくと、AIに作業を任せたときも規約に沿った対応をしてくれるので、ドキュメントと実装の乖離が起きにくくなりました。

この `CLAUDE.md` ＋ `.claude/rules/` の運用や、GitHub Pages × GitHub Actions で静的サイトを回す土台は、以前 Claude Code でコミュニティポータルサイトを作ったときの知見をそのまま引き継いでいます。

https://zenn.dev/easy_easy/articles/89318e4d0acb4f

## データの持ち方：単一ソースと「ブラウザは書かない」

可視化アプリで一番事故りやすいのが、保有データの二重管理です。画面用のデータと集計用の別データがずれて、表示が嘘をつく。これを設計で潰しました。

### 保有構成の単一ソースは取引台帳CSV

```text
transactions.csv ─┐
                  ├─ import-holdings.mjs ─→ holdings.json / js/data.js
prices.csv ───────┘                              │
   ↑ update-prices.mjs（株価自動）                 └─ アプリ・snapshot.mjs が参照
```

- **`data/transactions.csv`** … 1行＝1取引（`buy`/`sell`・株数・約定単価）。追記のみで運用し、過去行は消しません。`import-holdings.mjs` が移動平均法で株数と加重平均取得単価を畳み込み、全売却（残0）は自動除外します。保有銘柄・保有量・取得単価はここが唯一の正。
- **`data/prices.csv`** … 時価のスナップショット。国内株・米国株は日次cronが Yahoo Finance から自動更新します。手動更新が要るのは金・為替・投信の行だけです。
- **`data/holdings.json`** … アプリが読む保有データですが、上記2つのCSVから生成される派生物です。`js/data.js` の既定データも同時生成して二重管理を防ぎます。直接編集は禁止（pre-commit がCSVとの一致を検査）。
- **`data/history.json`** … 評価額の日次スナップショット履歴。GitHub Actions だけが書き込み、毎日 22:00 JST にコミットします。推移グラフと全体の前日比/前月比の実測値に使います。

### ブラウザは一切リポジトリに書き込まない

閲覧者の端末で取れる為替・金スポットはライブ取得し、共有履歴がない場合のフォールバックとしてのみ localStorage に蓄積します。サーバもDBも持たないので、データの真実はGitリポジトリとCIに閉じます。これが静的サイトで動的に見せるための肝で、運用の単純さとプライバシーを同時に確保できました。

## 株価の自動取得：yfinance から Yahoo の chart エンドポイントへ

当初は時価も手で `prices.csv` に書いていました。でも銘柄ごとの前月比まで出したくなった時点で、公開株価の推移なんだからどこかから自動で取れるはず、と Claude Code に相談しました。

最初に候補に挙げたのは Python の `yfinance` ライブラリです。実績もあり手堅い。ただ実装方針を詰めるうちに、CIに Python ランタイムと pip 依存を一つ増やすのが、このプロジェクトの「依存ゼロ・ビルドレス」志向と噛み合わないと気づきました。そこで最終的に、`yfinance` が内部で叩いているのと同じ Yahoo Finance の chart エンドポイント（`query1.finance.yahoo.com/v8/finance/chart/{symbol}`）を、Node のスクリプトから直接叩く形に落としました。ライブラリは使わず、考え方だけ借りた格好です。

この設計の良かった点がいくつかあります。

- `range=3mo` の1リクエストで、最新終値・前営業日・約1か月前終値を同時に取れます。だから過去分を別保存せずステートレスに回せます（前日比・前月比がこれだけで揃う）。
- 非公式エンドポイントはCIに隔離しました。ブラウザから叩くとCORSに当たるうえ、提供元の都合で壊れる可能性もある。だから取得は `update-prices.mjs`（CI）に閉じ、失敗時は `prices.csv` の既存値を維持する設計にしました。壊れても古い値のままで済み、画面が空になりません。
- 自動対象は国内株・米国株だけ。金・為替・投信は性質が違うので手動のまま、と割り切りました。

便利そうなライブラリを入れる前に、それが土台の思想と合うかを一度立ち止まって考える。長く使う道具では、この判断が効いてきます。

## 機能を見ながら育てる

ここからが本記事の中心です。Claude Code との対話で機能を育てていったセッションで、何を考えて手を入れたかを順に残します。今回は足すより、やめる・整えるに時間を使いました。手を動かす動機の多くも、実機で見たら気になった、という具体的な違和感からでした。

「何となくこんな感じで」というざっくりしたイメージを伝えれば、AIがそれっぽい形にしてくれる時代にはなりました。とはいえ、やっぱり実際に触ってみて、ああでもないこうでもないと試行錯誤して、AIとの壁打ちも挟みながら、結果的に「自分にとってはこれが使いやすい」というかたちにたどり着くのだと思います。

今回のプロダクトは、まず自分だけが使うもの、自分の資産ポートフォリオを公開したいもの、という名目がはっきりしていたぶん、そこに集中できました。これが「多くの人にも使えるように」「いろんなニーズを取り込んで」となっていたら、まったく違う開発スタイルになっていた気がします。何のために作るのか、誰に向けたものか、何を達成したいのか、ここだけは譲れないというこだわり。AIが多くを肩代わりしてくれる時代になっても、その芯の部分は結局、作り手の中から出てくるのだろうなと感じました。

### 通底するこだわり：ファーストビューにドーナツを

振り返ると、序盤の指示の多くがひとつのこだわりから出ていました。スマホで開いた瞬間に、ポートフォリオ全体のドーナツが見えてほしい、という。

- 「評価額合計の表示領域をもう少しコンパクトにしたい」
- 「評価額の推移もコンパクトにして、ドーナツがファーストビューで見えるように」
- 「評価額合計と推移は情報が重複してるので、統合しちゃっても良さそう」
- モバイルで `評価損益／前日比／前月比` が縦に積まれて縦幅を食うのが気になる → サマリー見出しを圧縮

一目で全体を掴むという、このツールの存在理由を守るための調整でした。後で出てくる損益率ランキングの「モバイルで折りたたみ」も、元をたどればここに行き着きます。何を最初に見せるかは、機能の数より大事だと思っています。

### まず表示漏れを直す

コードを精査して見つけたのは、新機能ではなく作りかけの取りこぼしでした。推移グラフのロジック（`chart.js`）は選択期間の騰落額・率を計算しているのに、テンプレートが描画していなかったのです。

1年/3年/全期間を切り替えても、その期間でいくら増えたかが出ない。計算済みのものが画面に出ていないのは、新機能より優先度が高い。期間ラベルと連動する期間リターン行を1行追加し、始点→終点の騰落を騰落色で表示するようにしました。

### 推移グラフの泥臭い作り込み

推移グラフは、実機で見るたびに細かい不満が出て、いちばん往復した部分です。

- **丸ポチが楕円に伸びる** … SVGを横全幅に引き伸ばす（`preserveAspectRatio=none`）と、各点の丸ポチまで横に間延びして楕円になる。これは点だけ SVG をやめ、`%` 配置のHTMLオーバーレイ（真円）で描いて解決しました。横長でも円は円のまま。
- **任意地点の値が見たい** … 月ごとの点にカーソルを当てる／タップで、その時点の評価額が出るように。透明な操作レイヤーで最寄り点を拾い、強調ドット＋ツールチップを出します。
- **点が多すぎて潰れる** … 過去データを足したら点が増えすぎたので、狭い画面では常設ドットを省略してラインだけにしました。ただしドットが無くてもホバー/タップで値は出るようにして、情報は失わないように。
- **軸ラベルの本数と表記** … 横幅に応じて月ラベルの本数を増減（広いほど多く）、年をまたぐときは `YYYY/M`・同年内は `M/D`。全期間で見たとき何月か分かりづらい、という不満への対応です。

ちなみに過去データは、自分が実際に記録していた月次の資産額をそのまま投入しました（2021〜2026年ぶん）。データが増えたタイミングで、期間トグルを `1M/3M/6M` から `1年/3年/5年/全期間` に作り替えています。グラフは入れる中身が育つと、見せ方の正解も変わるんですね。

### トグルの位置がずれる問題

地味だけど粘ったのが、サマリーカードのトグル配置です。`評価額 ⇄ 資産構成` のモード切替を持たせたところ、モードを切り替えるとボタンの位置が左右に動く。原因は、隣に並ぶ期間トグルのボタン数が違う（評価額=4個／資産構成=2個）ことでした。

- 固定幅にする → 資産構成側でスペースが空くのが気になる
- ならばモードボタンをカード右端に固定し、ボタン数が変わる期間トグルを左方向に伸縮させる → PCは解決
- ところがスマホで折り返すと左寄せに戻り、同じ問題が再発 → `margin-left:auto` で右寄せを担保して決着

実機で見ないと気づけない類の不満で、PCで直すとモバイルで再発する、を何度か繰り返しました。PCの正解とモバイルの正解は違う、を地で行く例でした。

### 既存データからインサイトを引き出す

新しいデータを増やさず、今あるデータから意味を引き出す方針で機能を選びました。

- **評価損益率ランキング** … 上位3／下位3銘柄をカードで可視化。表を目視でスキャンせずに勝ち負けが掴めます。スマホでは既定で折りたたみにして、ファーストビューのドーナツを守りました（例のこだわりです）。
- **銘柄テーブルのソート** … 当初は並び替えバーを新設したのですが、「ヘッダのラベル自体をクリックして並び替えたい」と方向転換しました。デスクトップはヘッダクリック＋昇降トグル（矢印 ↑/↓、非選択は薄い ↕）、モバイルはヘッダがないのでチップ列で代替。データ無（—）の銘柄は方向に関わらず常に末尾、という細部まで詰めています。

UIは新しい部品を足すより、既存の部品に機能を持たせるほうが学習コストが低い。この方向転換はその好例でした。

### 証券サービスらしさを物差しにする

銘柄テーブルのカラム順では、現在値の置き場所を何度か行き来しました。銘柄名の下にティッカー・数量とまとめてみたり、独立カラムに戻したり。最後は「一般的な証券サービスで見る並びと比べて違和感がないか」を物差しに、Claude Code と一緒に最終形を詰めています。自分の好みだけでなく、利用者が他で見慣れた型に寄せる、という判断軸です。

### 作ってからやめた機能：共有モード

これはもともと、Claude Design に現状のダッシュボードを精査してもらって「足したり変えたりすると良さそうな候補」を挙げてもらった中のひとつでした。Claude Design はプロトタイプを作って終わりではなく、いまの画面を見たうえで改善案を並べてくれるので、その候補を実際に試してみた、という位置づけです。

公開ダッシュボードなので、金額を伏せて構成と成績だけ見せる共有モード。整形層（`format.js`）で金額・数量を `¥•••` にマスクし、比率・騰落率・銘柄名・現在値だけ残す方式です。マスクを整形層に集約したので、全画面に自動で効きました。

ところが実物を見て、これはそもそも「自分の資産ポートフォリオをオープンに公開したい」という目的から外れていることに気づきました。肝心の金額がマスクされると、結局は割合くらいのふわっとした情報しか見えない。それって、まさに自分が既存のポートフォリオ紹介に物足りなさを感じていた状態そのものです。だったらこれを作る意味がないな、となって撤去しました。

金額を伏せて共有する仕組み自体は、家計簿アプリやSNSのスクショ共有などでもよく見かける、ごく一般的な機能ではあります。「大まかな構成は共有したいけれど、具体的な金額を出すのはためらう」というニーズがあるなら、共有モードは間違いなく効く。ただ今回はひとまず自分が使うだけ、しかも金額まで含めて公開するのが狙いだったので、よくある機能とはいえ、このプロダクトには必要ありませんでした。

これは失敗ではなく、実装して初めて要否が判断できる類のものだと思っています。整形層に隔離してあったおかげで撤去も一手で済みました。作る前に決めるより、安く作って見てから捨てるほうが速いこともある。関心の分離は、こういう撤退のときにも効きます。

今回の自分のケースでは不要でしたが、こうした候補をいろんな観点から出してくれること、そしてそれを実際の画面デザインのプレビューですぐ確認できることは、Claude Design の強みのひとつかもしれないと感じました。言葉で「こういう機能どう？」と議論するより、出てきた画面を見て「あ、これは違うな」と即断できるぶん、試して捨てるサイクルが速く回ります。

### いちばん粘った OGP 自動生成

SNSでシェアしたときに、その日の資産構成ドーナツと区分別の割合・損益が入ったカード画像を出したい。これが今回いちばんの作業でした。専用ブランチを切って、PRにしてマージ、という普通の開発フローでも回しています。

設計上の制約として、SNSクローラはJSを実行しません。だからOG画像は静的にコミットされたPNGである必要があり、クライアントでの動的生成は使えません。そこで次の流れを組みました。

1. `og.html` … `data/history.json` を読んでドーナツ＋凡例を 1200×630 で描く独立ページ
2. `scripts/og.mjs` … CIで Playwright が `og.html` を実ブラウザで開き、その領域をキャプチャして `og.png` を書き出す
3. `snapshot.yml`（日次cron）に組み込み、履歴更新と同時にOG画像も再生成・コミット
4. `dashboard.dc.html` の `og:image`／`twitter:image` を Pages のサブパスを含む絶対URLで指定（meta値は絶対URLが正）

試行錯誤のポイントはこんな感じでした。

- **フォント** … 当初のキャプチャでフォントが変わってしまいました。`og.html` 側で Google Fonts（Manrope / Noto Sans JP）を読み、Playwright で `document.fonts.ready` を待ってから撮ることで、ダッシュボードと同じ字形に揃えました。
- **ドーナツ中央の損益表示** … サイト本体は損益額と損益%を2行に分け、日本式（プラス＝赤）で着色しています。OG側が1行・緑になっていたので、本体に合わせて作り直しました。画像とサイトで表現が食い違わないことを優先しています。
- **凡例レイアウトのはみ出し** … OGは凡例幅が限られるため、損益が右にはみ出てクリップされる事故が起きました。バー幅・フォント・ドーナツ径・余白を実測しながら詰めて収めています。割合バーが構成名と離れすぎる、本文が上に寄って下に余白が出る、といった細かいズレも、実表示と揃うまで調整しました。
- **シェア検証で出たSEO警告** … `opengraph.xyz` で確認したら「タイトルが短い」「description が短い」「`og:site_name` が無い」と警告が出たので、タイトル・説明文・`og:site_name` を補いました。実際にプレビューしてはじめて分かる指摘です。

CIで実ブラウザを起動してスクショを撮るという一見重いやり方を取ったのは、サイトと寸分違わぬ画像を保証する確実な方法だからです。Canvasで描き直す手もありましたが、字形やレイアウトがサイトと乖離するリスクを嫌いました。

### 凡例レイアウトの長い試行錯誤

論点のひとつが、凡例の「バー＋％」が同じ構成比を二重に表現していることでした。ここから、実物を見ては直す往復を何度も重ねました。志向がいちばん出た部分なので詳しく残します。

1. **％をバーに統合** … 独立した％列を廃し、バー右端に小さく％を添える
2. **バー全廃→数値リスト化** … ドーナツ本体が既に構成比の視覚表現を担っているのだから、凡例のバーは役割が重複している、という気づき。％をカテゴリ色で着色してドーナツと呼応させた
3. **しかし色が衝突** … 構成比％をカテゴリ色にすると、スマホで色付きの評価損益（騰落色）と隣り合い、視認性が落ちた。色は意味にだけ、というルールを守るほど、意味の違う色が近接すると読みにくくなるジレンマ
4. **着地** … 構成比％は墨色に統一（色はスウォッチに任せる）。評価額は名称下のサブ表示から独立カラムに格上げ。並びを「構成名｜棒グラフ＋割合｜評価額｜評価損益」に整えた
5. **PC/モバイルで作り分け** … 最終的に、PC・タブレットは棒グラフ＋割合を損益と等幅で整列／モバイルはサブテキストをやめて割合・評価額をカラム表示、という形に落とした

ここでの学びは、情報を削るより役割を分けるほうが効く、ということでした。バーは視覚比較、数値は正確な値、色は分類。それぞれの担当を決めると、要素を減らさなくても画面が静かになります。そして色のルールは、単体で正しくても近接で破綻しうる。レスポンシブではPCの正解とモバイルの正解が違うという当たり前を、実機を見て初めて受け入れられました。

### 細部：ドーナツ中央の白縁

ドーナツの上に評価額合計が重なると、セグメントの色によっては数字が読みにくくなります。とくにOG画像で黒テキストとドーナツが被ると視認性が悪い。CSSの `text-shadow` で白い縁取りを金額と損益にだけ掛け、小さなラベルには掛けない、という最小限の処理で読みやすさを確保しました。地味ですが、毎回目に入る場所なので効きます。

## セキュリティ・公開範囲で気をつけたこと

資産情報を扱う公開サイトなので、ここは慎重に設計しました。ある回では Claude Code に「現状の実装にセキュリティ的な懸念がないか精査して」と棚卸しを頼み、出てきた指摘を一つずつ潰しています。

- **公開範囲の最小化** … `deploy.yml` は公開専用ディレクトリ `_site/` を組み立て、サイト動作に必要なファイルだけをコピーします。取引台帳CSV・スクリプト・`.claude/`・`.git/` は Pages に載せません。`dashboard.dc.html` は `index.html` に複製され、`<head>`（OGPメタタグ含む）がクローラに届きます。
- **サプライチェーン対策** … 外部 GitHub Actions は浮動タグではなく commit SHA でピン固定します（Action乗っ取り対策）。さらに Dependabot が週次でSHA更新PRを出すようにして、固定と追従を両立しました。
- **外部APIレスポンスの検証** … ブラウザから叩く為替・金スポットの応答を有限数値チェックで検証し、おかしな値が来たら props の代替値にフォールバックするよう堅牢化しました。
- **規約の自動検査** … 前述の pre-commit（`check-rules.mjs`）で、禁止語の直書きや絶対パス参照、ドキュメント未同期を検出します。
- **公開リスクの明示** … private リポジトリでも、GitHub Pages で公開したサイトURL自体は誰でも閲覧できます（アクセス制御付きの非公開 Pages は GitHub Enterprise Cloud のみ）。資産情報がURL経由で公開される前提で使う旨をREADMEに明記しました。なお取引台帳・ソースは `_site/` に含めないので、サイトからは取得できません。

## GitHub Actions による自動更新・デプロイ

触らず最新を実現しているのが GitHub Actions です。日次cron（`snapshot.yml`）で、株価更新 → 保有データ再生成 → 評価額スナップショット追記 → OG画像再生成 → コミット、までを自動で回します。デプロイ（`deploy.yml`）は `main` 更新時・snapshot 完了時（`workflow_run` の連鎖）に `_site/` を組み立てて GitHub Pages へ公開します。

設定で意識したポイントは次の通りです。

- **SHAピン留め**: すべてのActionsをコミットハッシュで固定し、サプライチェーン攻撃のリスクを軽減（更新は Dependabot に任せる）
- **permissions の最小化**: コミット＆プッシュ用・Pages デプロイ用に絞る
- **変更なし時のスキップ**: 差分がなければコミットせず、不要な空コミットを防止
- **OG画像の同時再生成**: 履歴更新とOG画像の再生成を同じワークフローで行い、画像とデータの鮮度を揃える

ブラウザは一切リポジトリに書き込まないので、自動化はすべてCI側に閉じます。閲覧者は常に最新のスナップショットを見るだけ、という構造です。

## 制作のワークフロー：Claude Design ⇄ ローカル ⇄ Claude Code

最後に、今回いちばん紹介したかった作り方のループです。

1. Claude Design で大枠のデザインと動くプロトタイプを作る（案A/B/C を見比べて方向を決める）
2. 受け取った変更を手元のリポジトリに適用・コミット・push して Git の正にする
3. Claude Code で実データ・CI・挙動調整を作り込む。実機（PC/スマホ）で見て気になった所を、対話で直しては確認、を繰り返す
4. 確定したものだけがコミットされ、また次の「見て直す」へ

このやり方の良いところは、実物を見て判断するのと、Gitで正とするのを両立できる点です。判断は常に動く画面に対して行い、確定したものだけが記録に残る。共有モードのように作って見てやめる判断も、このループだから速くできました。デザインの発散は Claude Design、実装の収束は Claude Code、と道具を使い分けた感覚です。

余談として、変更ファイルをZIPで受け取った際、Macの Finder で解凍すると「エラー14」が出てつまずきました。原因はZIP内のドットファイル／ドットフォルダ名で、`.github/` や `.claude/` をドット無しの名前に置き換えて出力し直すと解決しました。AIとの開発でも、こうした受け渡しまわりの泥臭い部分は地道に潰す必要があります。

## おわりに

振り返ると、設計判断のほとんどが同じ価値観から出ていました。

1. **メンテナンスコストを極力抑えること**（数年後の自分が無理なく維持できる状態） → ビルドレス、依存ゼロ、キー不要API
2. **データが嘘をつかないこと** → 取引台帳という単一ソース、生成物は直接編集しない、pre-commit で一致を強制
3. **触らず最新であること** → CIが株価取得・スナップショット・OG生成・デプロイを日次で回す
4. **意味のない色・数字・装飾を足さないこと** → 情報密度は上げつつ、ノイズは削る
5. **安く作って、見てから捨てること** → 関心の分離が、機能の撤退も楽にする

道具の使い分けについても少し。Claude Design は方向を決めるのが速いです。複数案を動く形で並べてくれるので、文字の仕様で悩むより先に「これだ」を選べる。トンマナと配色の軸が最初に固まると、その後の細部判断がぶれません。一方の Claude Code は、決めた方向を実機で見ながら詰めるのが強い。今回のように「スマホで見たら縦幅を食う」「中間幅でテキストがはみ出す」といった、実物を触らないと出てこない不満を、対話で直してはすぐ確認できます。発散と収束で道具を分けると、両方の良さが活きました。

もちろん、AIに任せきりにはできません。生成されたUIは実際にブラウザやスマホで触ると気になる点が必ず出てきます。凡例レイアウトの試行錯誤がまさにそうで、自分の目で確認して改善を重ねることが完成度につながりました。大量の変更時には意図しない上書きもあり得るので、`git diff` での差分確認も欠かせません。

それこそハーネスやループエンジニアリング的な設計を突き詰めていけば、作業のほとんどはAIに任せられるようになるのかもしれません。

それでも、最終的なアウトプットを見るのは自分だし、何か問題が起きればそれは自分の責任です。そういう責任や倫理にかかわる部分は、人間がやらざるを得ない。

というより、「安易に手放さずに人間がやるべきところ」だと、自分は思っています。

それでも、あくまで個人サービスなので「多くの人にとって最適」を目指す必要がなく、「いまの自分にとっての最適解」をすぐ形にできたのは良かったです。

あとから見て「やっぱりイマイチだな」と思う部分も出てくるのでしょうが、その改善サイクルを高速に回せること自体が、AIエージェントを軸にした開発の強みだと感じました。

開発ワークフローやCI、ハーネス、セキュリティ、公開の仕方といった土台の部分は、これまでAIで作ってきたいくつかの個人サービスの知見がそのまま活きています。

思いついたアイデアをすぐ形にできること、そして数を打てること。これは今までの自分のスキルだけでは届かなかったところなので、AIの恩恵を存分に受けた部分だと思います。

機能を増やすより、既にある価値を画面に出し、要素の役割を整え、運用を自動化する。地味ですが、長く使う個人の道具にはこれが効きます。資産の可視化ツールを作っていたつもりが、できあがったのは、手をかけずに正しくあり続ける仕組みでした。

複数口座に資産が散らばっている人、自分の「一枚で見たい」を持っている人は、Claude Design で大枠を作って Claude Code で詰める、というこのループを試してみると面白いと思います。
